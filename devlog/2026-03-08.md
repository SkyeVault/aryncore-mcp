# Devlog — March 8, 2026

**Project:** ArynCore MCP
**Session type:** Major build — local AI desktop hub

---

## What existed before this session

The project had a solid Python backend with no frontend at all. The `frontend/` directory was an empty placeholder. What was working:

- Five AI personas (Aryn, Doc, Kona, Glyph, Estra) backed by a remote Ollama server at `162.248.7.248:11434`, all using Mistral
- A CLI-based orchestrator (`backend/mcp_orchestrator.py`) to talk to personas from the terminal
- File-system job triggers that routed GPU tasks (SadTalker, Stable Diffusion) via Watchdog
- Docker services for n8n (port 5678), TortoiseTTS (port 5003), and Prometheus (port 9090)
- SadTalker, TortoiseTTS, Whisper, AnimateDiff installed but not integrated into any UI
- A setup doc for n8n webhooks that was never implemented

The system was headless and hard to use day-to-day. No way to manage models, switch servers, or trigger tools without the terminal.

---

## Goal for this session

Build a local-first desktop home app that pulls everything together:

- Chat with any LLM model across any server
- Manage Ollama models (browse, pull, delete) with a visual catalog
- Connect and manage multiple remote servers
- Integrate with n8n (local and networked)
- Run GPU tools (SadTalker, TTS, SD) from a UI
- Build visual workflows that chain tools together

## What was built

### FastAPI Backend

Created a full REST + WebSocket API layer in `backend/` to expose all existing functionality over HTTP.

**`backend/api.py`** — FastAPI app entry point with CORS, lifespan hooks, and router mounts.

**`backend/db.py`** — SQLite database via `aiosqlite`. Schema includes:
- `servers` — registry of Ollama/API servers with auth token support
- `conversations` + `messages` — persistent chat history
- `n8n_instances` — multiple n8n connections
- `settings` — key/value config store
- `workflows` — saved workflow graphs

Pre-seeded with Local Ollama (`localhost:11434`), Remote Ollama (`162.248.7.248:11434`), and Local n8n (`localhost:5678`).

**`backend/routers/system.py`** — System status endpoint. Pings all services concurrently (Ollama, n8n, TortoiseTTS, Stable Diffusion, Prometheus) and reads GPU stats via `nvidia-smi`. Returns live data including GPU name, temperature, VRAM usage, and utilization.

**`backend/routers/ollama.py`** — Ollama management:
- `GET /models` — list models on any server
- `GET /ping` — check if an Ollama server is reachable
- `POST /start-local` — start `ollama serve` if it isn't running, waits up to 8 seconds to confirm
- `POST /pull` — streaming model pull with pre-flight reachability check and proper error propagation
- `DELETE /model` — remove a model
- `GET /show` — model details

**`backend/routers/chat.py`** — WebSocket chat (`/api/chat/ws`) with streaming token delivery. Loads persona system prompts from `mcp/config/models.json`. Persists conversations and messages to SQLite. Supports mid-conversation model and server switching.

**`backend/routers/servers.py`** — CRUD for the server registry. Each server can be pinged individually. Supports `ollama`, `openai-compatible`, and `custom` types.

**`backend/routers/n8n.py`** — n8n integration. Lists instances, fetches workflow lists via n8n API, checks instance health, and triggers webhooks with arbitrary payloads.

**`backend/routers/tools.py`** — Tool registry. Checks each tool's status by pinging its port or checking if its script exists. Exposes run endpoints for Stable Diffusion (txt2img), TortoiseTTS (TTS), and SadTalker (file upload, subprocess).

### React Frontend

Initialized a Vite + React + TypeScript project in `frontend/` from scratch.

**Stack:** React 19, Vite 7, Tailwind CSS v4, Zustand, React Router v7, Lucide icons, ReactFlow, Radix UI primitives.

**`frontend/src/lib/api.ts`** — Typed API client covering all backend endpoints. All types defined here (Server, OllamaModel, Conversation, Message, N8nInstance, Tool).

**`frontend/src/lib/ws.ts`** — `ChatSocket` class. Manages WebSocket lifecycle, reconnection, and typed message dispatch.

**`frontend/src/store/useAppStore.ts`** — Zustand global store. Tracks active persona, active model, active server, conversations, server list, n8n instances, tool list, and system status. PERSONAS array defined here with names, roles, and colors.

### App Sections (7 pages)

**Dashboard (`/`)** — System health cards for all services, GPU widget (name, temp, VRAM, utilization), and persona quick-launch buttons that navigate directly into Chat with the selected persona pre-set. Status polling every 10 seconds.

**Chat (`/chat`)** — Full streaming chat with WebSocket. Left sidebar shows conversation history with delete per conversation, and a New Chat button. Main area streams tokens in real time with a blinking cursor while the model is responding. Persona and model are selected globally from the TopBar. Conversations and messages are saved to SQLite.

**Model Manager (`/models`)** — Two tabs:

- *Installed* — Lists all models on the selected server with size, quantization level, an `ACTIVE` badge if it's the current model, a **Chat** button (sets active + navigates to Chat), and a **Workflow** button (navigates to Builder and drops an LLM node pre-labelled with the model name).
- *Model Catalog* — 26 curated models across 6 categories (General, Code, Fast/Small, Large, Vision, Embeddings). Each card shows the model tag, size estimate, and a plain-English description. One-click Pull button with a live progress bar. After install, cards flip to show Chat and Workflow shortcuts.

**Servers (`/servers`)** — Add, test, and manage Ollama/API server connections. Each server card shows live ping status (green/red), host, port, and type. A "Use" button sets the active server for chat. Supports unlimited servers.

**n8n Hub (`/n8n`)** — Left panel lists n8n instances with online/offline indicator. Right panel shows quick-trigger buttons (`/blog`, `/video`, `/social`, `/img`), a custom webhook form with JSON payload editor and response display, and a workflow list pulled from the n8n API. Supports adding multiple n8n instances (local and networked). Direct "Open n8n" link per instance.

**Tool Library (`/tools`)** — Visual card grid for all GPU tools. Each card shows tool type (image/audio/video), status (online/offline/available/unavailable), and port. Clicking a card expands an inline runner panel:
- Stable Diffusion: prompt, negative prompt, steps slider → generates and displays base64 images
- TortoiseTTS: text + voice preset → sends to Tortoise API
- SadTalker: image + audio file upload → runs inference via subprocess

**Workflow Builder (`/builder`)** — ReactFlow canvas with a node palette sidebar. Pre-defined node types: Input, Output, LLM Chat, TortoiseTTS, SadTalker, Stable Diffusion, n8n Webhook. Drag to add, connect nodes visually. Save workflow as JSON. Simulated run that steps through nodes with a log. When navigated from Model Manager via the Workflow button, automatically drops the selected model as a pre-labelled LLM node.

## Bugs found and fixed during session

### Bug 1 — WebSocket "Connecting to backend" (Chat never worked)

**Cause:** The frontend WebSocket client was connecting to `/ws/chat/ws`. The Vite proxy only forwarded `/ws/*` paths with `ws: true`. But the FastAPI backend serves the WebSocket at `/api/chat/ws` (because the chat router is mounted under `/api/chat`). The path never matched, so the connection always failed.

**Fix:**
- Changed `ws.ts` to connect to `/api/chat/ws`
- Moved `ws: true` onto the `/api` proxy entry in `vite.config.ts` and removed the separate `/ws` proxy rule
- Required a dev server restart (vite.config.ts changes don't hot-reload)

### Bug 2 — Model pull hung silently / showed no feedback

**Cause 1:** Local Ollama was installed (`/usr/local/bin/ollama`) but `ollama serve` was never started. The backend pull used `timeout=None`, so it connected to a closed port and hung indefinitely with zero bytes streamed back. The frontend showed a frozen progress bar.

**Cause 2:** The pull error generator silently swallowed exceptions — if the connection failed, the `StreamingResponse` just ended with no data and no error message reaching the frontend.

**Cause 3:** The Model Catalog defaulted to pulling from `localhost:11434`, which was offline. No visual indication of this before attempting the pull.

**Fix:**
- Added `POST /api/ollama/start-local` endpoint — spawns `ollama serve` as a subprocess and polls until it responds
- Added `GET /api/ollama/ping` endpoint for lightweight reachability checks
- Added pre-flight check in `POST /pull`: if the server is unreachable, fail immediately with a clear 502 error instead of hanging
- Set a proper `httpx.Timeout` (10s connect, unlimited read) on the pull stream
- Error lines from the Ollama stream (`{"status": "error: ..."}`) are now propagated to the frontend and shown on the card
- Added `ServerBanner` component on the Model Manager page — shows green/red server status, and a **Start Ollama** button when local is offline. Button calls `/start-local`, waits for confirmation, then refreshes the model list.
- Model page now auto-selects the Remote Ollama server if it loads before local comes up
- Dismiss button on pull error cards

**Outcome:** Clicked Start Ollama from the UI — Ollama started successfully. `qwen2.5-coder:14b` (14.8B, Q4_K_M) was already installed and appeared in the model list.
